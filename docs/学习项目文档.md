# 学习项目文档

## 1. 这份文档的目标

这份文档只围绕 `draft.txt` 来讲，不展开别的主题。

你可以把它理解成一份“对照学习手册”：

- `draft.txt` 说这个项目有什么特点
- 这个仓库里是怎么把这个特点落地成最小实现的
- 你在项目里应该先看哪里
- 运行时你能怎么直观感受到这个特点
- 你后面如果自己改，会怎么用到它

换句话说，这份文档不是在讲“怎么做一个旅游网站”，而是在讲：

`draft.txt` 里的那套 `FastAPI + LangGraph + LangChain + MCP + 多 Agent + Plan/Execute/Replan + ReAct + 缓存 + interrupt + 上下文压缩`

在这个学习项目里是怎么写出来的。

## 2. 先把 `draft.txt` 压成项目特点

`draft.txt` 的项目介绍，核心可以拆成下面 9 个项目特点：

1. 技术栈是 `Python | FastAPI | LangGraph | LangChain | MCP | LLM`
2. 项目目标是生成覆盖 `住宿 / 交通 / 天气 / 景点 / 美食` 的旅游攻略
3. 工作流是 `Plan -> Execute -> Replan`
4. 架构上是 `1 个父 Agent + 6 个专业子 Agent`
5. `Execute` 阶段是 ReAct 风格循环
6. 工具层是 6 类 MCP 服务
7. 工具结果要做缓存复用
8. 用户反馈时要做局部重规划，而不是整份重跑
9. 信息不足时要暂停提问，同时做上下文管理

这个仓库就是按这 9 个特点去复现的。

## 5. 整体执行链路

先用一句话把主链路记住：

`用户请求 -> FastAPI -> LangGraph -> 父 Agent 拆任务 -> 子 Agent 调 MCP -> 汇总攻略 -> 如有反馈则局部重规划`

更具体一点：

1. 用户提交 `POST /plan`
2. `TravelAssistantService` 创建一份 LangGraph 初始状态
3. `ensure_requirements` 判断信息够不够，不够就 `interrupt`
4. `plan_tasks` 让父 Agent 生成 6 个专业任务
5. `execute_tasks` 逐个交给子 Agent 执行
6. 子 Agent 先查缓存，再调 MCP，再检查结果是否完整
7. `compose_guide` 把结果收拢成最终攻略
8. 如果用户不满意，调用 `/feedback` 只重算受影响部分

这条链路是后面所有“项目特点”的公共背景。

### 特点 1：攻略输出不是一段大模型闲聊，而是固定结构的旅游规划

#### 这在 `draft.txt` 里是什么意思

`draft.txt` 明确说，最终要覆盖：

- 住宿
- 交通
- 天气
- 景点
- 美食

这说明它不是做一个“聊天回答器”，而是做一个有固定输出结构的规划系统。

#### 为什么这个项目要这么写

如果直接让一个大模型随便输出长文，你很难做到：

- 让每个领域对应一个专业工具
- 在用户反馈时只改其中一部分
- 把缓存和工具结果稳定复用起来

所以这里先把最终输出拆成固定分区，整个系统才有清晰的结构边界。

- [app/core/llm.py](/D:/SomeFunnyProjFromGithub/foroffer/app/core/llm.py) 里的 `compose_guide_markdown`

#### 运行时怎么直观看到

1. `POST /plan`
2. `POST /plan/{session_id}/resume`

生成的攻略会稳定分成：

- `交通`
- `住宿`
- `天气`
- `景点`
- `美食`

这就是“结构化攻略输出”。

#### 你以后会怎么用

如果你后面想加真实工具或真实模型，这个结构会成为你的稳定输出协议。工具换了，模型换了，分区结构最好别轻易变。

#### 学习路径

1. 先运行项目，生成一次完整攻略。

### 特点 2：工作流是 `Plan -> Execute -> Replan`

#### 这在 `draft.txt` 里是什么意思

它不是“收到请求立刻调工具”，而是分成 3 段：

1. `Plan`
2. `Execute`
3. `Replan`

这意味着系统先决定做什么，再执行，再汇总与调整。

#### 为什么这个项目要这么写

如果没有 `Plan`，工具调用会变成混乱堆叠。

如果没有 `Replan`，你拿到的只是一堆散乱工具结果，而不是一份攻略。

所以这 3 段的作用分别是：

- `Plan`：把问题拆成任务
- `Execute`：让子 Agent 执行任务
- `Replan`：把执行结果整理成最终输出

#### 先看哪里

- [app/graph/workflow.py](/D:/SomeFunnyProjFromGithub/foroffer/app/graph/workflow.py)

特别注意 4 个节点：

- `ensure_requirements`
- `plan_tasks`
- `execute_tasks`
- `compose_guide`

其中真正对应 `Plan -> Execute -> Replan` 的是后 3 个。

#### 运行时怎么直观看到

你可以把它理解成 3 次状态变化：

1. `planned`
2. `executed`
3. `completed`

虽然页面不会把每一步细节展开给你，但图里就是按这个顺序执行的。

#### 你以后会怎么用

以后如果接真实工具，这个结构很有用。你可以单独替换 `Plan` 逻辑，也可以单独优化 `Execute`，而不用把整套系统一起改烂。

#### 学习路径

1. 先打开 `workflow.py`，只看节点名和边，不看实现细节。
2. 再看 `plan_tasks`，理解父 Agent 负责“拆任务”。
3. 再看 `execute_tasks`，理解这里为什么只负责“跑任务”，不负责最终文案。
4. 最后看 `compose_guide`，理解为什么它叫 `Replan` 风格的整理，而不是简单字符串拼接。

### 特点 6：缓存复用不是附属功能，而是工作流的一部分

#### 这在 `draft.txt` 里是什么意思

它强调“调用前基于历史执行结果进行检索；命中相同查询时直接复用”。

#### 为什么这个项目要这么写

多 Agent + 多工具最怕的就是重复查一样的东西。

如果没有缓存：

- API 成本会更高
- 响应会更慢
- 反馈重规划时也会重复浪费调用

### 特点 9：上下文管理是为了让多 Agent 工作流可持续运行

#### 为什么这个项目要这么写

对于工具型 Agent 来说：

- `ToolMessage` 往往比闲聊更有价值
- 最近几轮用户意图也必须保留

但如果全部保留，图会越来越重。

所以这里做的不是“保留所有历史”，而是“保留最有用的历史”。

2. `Plan -> Execute -> Replan`
   这是整个架构主线
3. 反馈时局部重规划
   这是最能体现“不是普通聊天机器人”的地方
4. 父 Agent + 6 个子 Agent
   这是理解整个拆分方式的核心

你先把这 4 个看明白，再看缓存和上下文，会顺很多。

## 8. 最推荐的实操顺序

如果你要真正把这个项目吃透，建议按下面顺序学：

1. 跑一次“缺信息 -> 补信息 -> 出攻略”
2. 跑一次“住宿想更便宜一点”的反馈
3. 看 `workflow.py`，建立主流程地图
4. 看 `worker.py`，理解 ReAct 执行器
5. 看 `mock_servers.py`，理解 MCP 服务边界
6. 看 `cache.py` 和 `context.py`，理解系统为什么能控制成本和复杂度
7. 最后再去改 `llm.py` 和 `parsing.py`

这个顺序的重点是：

先从“行为”入手，再回到“结构”，最后再改“策略”。

## 9. 一句话总结

这个学习项目最值得学的，不是“它能生成一篇旅游攻略”，而是：

它把 `draft.txt` 里那套多 Agent、分阶段、可中断、可复用、可局部重规划的系统思路，压成了一份你能直接跑、直接看、直接改的最小实现。
